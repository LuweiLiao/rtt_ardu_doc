# RTT ArduPilot 性能瓶颈分析 — 263Hz → 400Hz 路线图

> 本文件记录通过 ChibiOS 参考精读 + RTT 当前实现对比 + 硬件资源审计，
> 系统量化每个优化方案的预计收益，避免"猜瓶颈"式低效优化。
>
> 2026-05-14 新增：P1 SPI DMA 定量分析结论 — 收益仅 ~4%

## 基线测量

### V1 基线 (P0 ADC DMA + EOC 轮询 1kHz)

| 指标 | 值 |
|------|-----|
| 循环率 | ~263 Hz (3800 µs/loop) |
| CPU 空闲 | ~47% (loop中约 1800 µs 空闲) |
| 目标 | 400 Hz (2500 µs/loop) |
| 需节省 | **~1300 µs/loop** |

### V2 基线 (P0 ADC DMA + 100Hz gate + EOC 轮询, 2026-05-14 实际测量)

| 指标 | 值 |
|------|-----|
| 循环率 | **~1387 Hz** (721 µs/loop) |
| 测量 | 60s → 83,242 次迭代 |
| 关键改动 | `_timer_tick` 从 1kHz → 100Hz (`AP_HAL::micros()` gate) |

> ⚠️ **关键发现：100Hz gate 带来的收益远超 ADC DMA 本身。** 原始 1kHz ADC 轮询消耗了 ~8% CPU（80 µs × 1000 Hz = 80 ms/s = 8%）。降频到 100Hz 后 ADC 开销降到 ~0.08% CPU。

## 瓶颈分解

### 1. SPI1 轮询 (IMU 总线)

**当前实现** (`SPIDevice.cpp:173-262` `spi1_poll_transfer`):
- 逐字节：等待 TXE → 写 DR → 等待 RXNE → 读 DR
- 每字节 ~3 µs (@ 8 MHz + 轮询开销)
- 每次 IMU 读取 ~20 字节 → ~60 µs
- 每 loop 2-3 IMU 传感器 → **~120-180 µs**

**DMA 方案** (ChibiOS SPIDevice.cpp:191-209 参考):
- `spiStartExchangeI()` 启动 DMA 传输
- `osalThreadSuspendTimeoutS()` 等待完成
- DMA 设置 ~5 µs，传输中 CPU 空闲
- 预计每 loop **~15 µs**
- **节省: ~105-165 µs/loop**

**收益: 263 Hz → ~275 Hz (≈+4%)** ❌ 远不足以达到 400 Hz

### 2. ADC 轮询 (已通过 P0 100Hz gate + DMA 修复)

P0 前：每 loop ~150 µs 8 通道轮询 + 定时器 1kHz 开销 80 ms/s
P0 后：DMA 连续扫描 + 100Hz gate → ~0 µs/loop 额外开销

**实际收益复盘**：
- **100Hz gate 贡献 ~80%** — 将 `_timer_tick` ADC 处理从 1kHz 降到 100Hz，CPU 开销从 8% 降到 0.08%
- **DMA 贡献 <20%** — 消除每 10ms 的 ~80 µs 轮询，但 100Hz 下轮询本身仅 0.08% CPU
- **100Hz gate 的实现**：`AP_HAL::micros()` + `_last_timer_tick` 判断 ≥10ms 才执行

✅ 已解决，总收益使循环率从 263 Hz 提升到 ~1387 Hz（远超目标 400 Hz）

### 3. 定时器线程抢占

**现状**: 定时器线程 1 kHz 优先级 4，主循环优先级 5
- 每 loop 被抢占约 2-3 次 (1000 µs tick)
- 每次抢占 ~50-100 µs (上下文切换 + 定时器回调)
- 每 loop ~100-300 µs 被抢

**方案 A — 降低定时器频率**: 1 kHz → 500 Hz
- 减少约一半抢占
- 预计节省: **~50-150 µs/loop**
- ⚠️ 风险：定时器精度降低 (GPS PPS, RC 输入)

**方案 B — 微调优先级**: 主循环 prio 4, 定时器 prio 3
- 减少定时器对主循环的抢占
- 预计节省: **~30-80 µs/loop**

### 4. delay_microseconds_boost 策略

**现状**: DWT 忙等上限 200 µs，忙等期间不释放 CPU

**优化**: 将上限降到 100 µs，≥100 µs 的延迟改为 rt_thread_delay
- 预计节省: **~50-100 µs/loop** (减少忙等累积)

### 5. SPI4 (MS5611) 轮询

**现状**: 通过 `rt_spi_transfer_message()` 多字节传输
- 气压计读取频率很低 (10 Hz)，不是循环率瓶颈
- **不计入** 263 Hz → 400 Hz 路径

## 优先级排列

| 优先级 | 优化项 | 预计收益 | 实施难度 | 实际结果 |
|--------|--------|----------|----------|----------|
| **P0** ✅ | ADC DMA (DMA+ISR) | ~150 µs | 中 | ⚠️ 需后续验证 |
| **P0.1** ✅ | 100Hz gate (_timer_tick) | **~700+ µs** (实际: 263→1387 Hz) | 低 | ✅ 已验证 |
| **P1** | 测量：定时器实际抢占时间 | — | 低 (数据采集) | |
| **P2** | DWT 忙等上限 100 µs | ~50-100 µs | 低 | |
| **P3** | 定时器频率 1k → 500 Hz | ~50-150 µs | 中 (风险: GPS/RC) | |
| **P4** | SPI1 DMA | ~105-165 µs | 高 (DMA+ISR+Sem) | |
| **放弃** | SPI4 DMA | 冲突无收益 | — | |

## 关键结论（2026-05-14 更新）

1. **SPI DMA 不是优先优化项** — 收益仅 ~4%
2. **ADC 100Hz gate 是最大惊喜** — 一项纯软件优化使循环率远超 400 Hz 目标
3. **400 Hz 目标已超额完成** — 1387 Hz 远超需求，进一步优化应聚焦其他领域
4. **如果 V3 基线+P DMA 也通过**，RTT 移植的循环率瓶颈实际上已被 100Hz gate 解决

## 方法论原则

1. **先测量，后优化** — 任何优化决策前必须量化当前值和目标值的差距分布
2. **用 ChibiOS 参考做基准** — 如果 ChibiOS 在同样硬件上可达 400 Hz，差距在调度/RTT OS 层
3. **收益/复杂度排序** — 选收益大复杂度低的优先
4. **不做贡献 <5% 的优化** — 除非是实现其他优化的前提
